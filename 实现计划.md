# 信息智能分析系统（InformationSmartAnalysisSystem）实现计划

## 1. 目标与范围

构建一个可针对给定信息源进行智能分析的系统。核心能力（来自《项目提示词.md》）：

1. 支持添加不同种类的信息源（官方网站 / 指定本地文件夹 / FreshRSS 指定源）。
2. 支持创建分析任务，每个任务可绑定多个信息源作为信息来源范围。
3. 每个任务可展示所绑定信息源的状态情况。
4. 每个任务可触发分析，支持基于信息源新增内容的**增量分析**；分析调用大模型接口（OpenAI 兼容，可配置）。

同时必须包含 CLAUDE.md 规定的基础模块：登录、权限管理、系统配置、任务中心。

> 关键决策（已与用户确认）：LLM 采用 **OpenAI 兼容接口（可配置 base_url/api_key/model）**；FreshRSS **已有实例待接入**，通过 Google Reader API 接入。

## 2. 技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 后端 | Python 3.11 + FastAPI + Uvicorn | 异步 API + 后台线程池执行任务 |
| 数据库 | SQLite（SQLAlchemy 2.x ORM） | 轻量，符合 CLAUDE.md 要求 |
| 鉴权 | JWT（python-jose）+ bcrypt | password.txt 为用户源，启动/登录时同步到 DB |
| 前端 | Vue 3 + Vite + TS + Element Plus + Pinia + Vue Router | 构建产物由 FastAPI 托管为静态资源，单服务部署 |
| HTTP 客户端 | httpx | 调用 web_fetch 服务 / FreshRSS / LLM |
| 文档解析 | PyMuPDF(pdf)、python-docx(docx)、原生(txt/md/html) | 本地文件夹信息源用 |
| 任务执行 | `concurrent.futures.ThreadPoolExecutor` + FastAPI lifespan | 进程内后台 worker，DB 记录状态 |
| 日志 | logging + TimedRotatingFileHandler | 按天切割到 logs/ |
| 测试 | pytest + httpx TestClient + monkeypatch | 单元 + 冒烟 |

外部依赖：
- **网页抓取**：集中式 web_fetch 服务（`http://192.168.0.111:33333`，API Key 见 CLAUDE.md），仅在 `config/app.json` 中配置，不硬编码。
- **LLM**：OpenAI 兼容接口（`config/app.json` 中 `llm.base_url/api_key/model`）。
- **FreshRSS**：Google Reader API（每个 FreshRSS 信息源自带 base_url/账号/token）。

## 5. 数据库设计（SQLite）

| 表 | 关键字段 | 说明 |
|---|---|---|
| `users` | id, username, password_hash, role(admin/user), created_at | 从 password.txt 同步 |
| `page_permissions` | id, user_id, page_key | 普通用户可见页面；admin 默认全部 |
| `info_sources` | id, name, type(website/local_folder/freshrss), config(JSON), status(ok/warning/error), last_sync_at, last_error, item_count, created_at, updated_at | 信息源 |
| `info_items` | id, source_id, external_id, title, url, content, content_hash, published_at, fetched_at, analyzed | 抓取到的单条信息，content_hash 去重 |
| `analysis_tasks` | id, name, description, config(JSON: 模式/提示词/模型覆盖), status, created_at, updated_at | 分析任务 |
| `task_sources` | id, task_id, source_id, last_analyzed_item_id, last_analyzed_at | **增量水位线**：按 (任务,源) 记录已分析到哪里 |
| `task_runs` | id, task_id, mode(full/incremental), status(pending/running/succeeded/failed), started_at, finished_at, summary, error | 一次分析执行 |
| `analysis_results` | id, task_run_id, task_id, source_id, info_item_id(可空), result_type(per_item/aggregate), content(TEXT/JSON), created_at | 分析产出 |
| `task_logs` | id, task_run_id(可空), level, message, created_at | 任务日志（任务中心展示） |
| `system_runtime` | key, value | 运行时键值（如服务启动时间） |

时间字段一律存 UTC，接口/前端展示转北京时间（CLAUDE.md 规范2）。

## 6. 后端模块设计

### 6.1 认证与权限
- `security.py`：启动时与每次登录时解析 `password.txt`（格式 `username:password:role`），用 bcrypt 哈希后 upsert 到 `users` 表。
- 登录返回 JWT；前端存于内存/Pinia，请求头带 `Authorization: Bearer`。
- `deps.py`：`get_current_user` 解析 JWT；`require_page(page_key)` 校验该用户是否有该页权限（admin 全放行；user 查 `page_permissions`）。
- 页面权限键：`dashboard / info_sources / analysis_tasks / task_center / permission / system_config`。

### 6.2 信息源管理与三类适配器
统一抽象接口（`base.py`）：
```
class InfoSourceAdapter:
    def check_status(self) -> SourceStatus        # 健康/计数/最近错误
    def fetch_new_items(self, since=None) -> list[InfoItem]  # 抓取新内容
```

- **website（官方网站）**：config={url, link_selector, content_selector, fetch_interval_min}。调用 web_fetch 服务获取 HTML -> 解析文章链接 -> 对新链接（按 url 哈希去重）抓取正文存 `info_items`。原始 HTML 可选落 `data/downloads/YYYY/MM/DD/`。
- **local_folder（本地文件夹）**：config={folder_path(绝对路径,可配置), patterns, recursive}。扫描目录，按 路径+mtime 识别新增/变更文件，抽取文本（PyMuPDF/python-docx/原生），content_hash 去重。
- **freshrss**：config={base_url, user, api_token, stream(feed id 或 label), mark_as_read}。走 Google Reader API：`GET {base_url}/api/greader.php/reader/api/0/stream/contents/{stream}?ot={since}`，取条目，可选抓取后 `mark-all-as-read`。content_hash 去重。

信息源状态：每次 `check_status`/`fetch_new_items` 更新 `info_sources.status/last_sync_at/last_error/item_count`，供任务页展示。

### 6.3 分析任务与增量分析引擎
`analysis/engine.py` 编排一次 `task_run`：
1. 读取任务绑定的源（`task_sources`）。
2. 对每个源：
   - **增量**：`fetch_new_items(since=task_sources.last_analyzed_item_id 对应的 published_at/fetched_at)`；只分析新条目。
   - **全量**：分析该源全部 `info_items`（水位线重置）。
3. 按任务 `config.mode` 调用 LLM：
   - `per_item`：逐条分析，存 `analysis_results(result_type=per_item)`。
   - `aggregate`：把本批新条目拼接送 LLM 汇总，存一条 `aggregate` 结果。
4. 推进 `task_sources.last_analyzed_*` 水位线（仅增量且成功时）。
5. 全程写 `task_logs` + 文件日志；更新 `task_runs.status/summary`。

`llm_client.py`：OpenAI 兼容（`POST {base_url}/chat/completions`，Bearer api_key），超时/重试/错误落日志。提示词模板放 `prompts.py` 并可在任务 config 中覆盖。

### 6.4 任务中心 / 系统配置
- **任务中心**：聚合 `task_runs`（信息源同步、分析任务执行都产生 run）+ `task_logs`，展示列表/状态/日志详情/过滤。
- **系统配置**：只读展示 `config/app.json`（脱敏密钥），并展示运行时信息（服务启动时间、DB 路径、PID）。

### 6.5 后台 worker
`worker.py`：进程级 `ThreadPoolExecutor`。API 触发分析 -> 创建 `task_runs(status=pending)` -> 提交到线程池 -> worker 执行引擎 -> 更新状态/日志。前端轮询任务中心查看进度。

## 8. 关键流程：增量分析

```
用户在"分析任务"页点击【增量分析】
  └─► POST /api/analysis-tasks/{id}/run {mode:"incremental"}
        └─► 创建 task_runs(pending) -> 提交 worker
              └─► engine.run(task, mode=incremental)
                    for source in task.sources:
                        new_items = adapter.fetch_new_items(since=watermark)
                        if mode per_item:  LLM 分析每条 -> analysis_results
                        if aggregate:      LLM 汇总本批   -> analysis_results
                        推进 task_sources 水位线
                    task_runs -> succeeded + summary
              └─► 全程 task_logs + 文件日志
前端轮询任务中心 / 结果页查看
```

## 10. 部署与运维

- **首次部署**：脚本检查并创建 `data/`、`data/downloads/`、`logs/`，写入默认 `password.txt`（内容见 CLAUDE.md 附件），初始化 SQLite。增量部署跳过创建。
- **start.ps1 / start.sh**：检查 venv（无则创建+装依赖）-> 构建前端（或用预构建 dist）-> 后台启动 uvicorn -> 写 `logs/server.pid`。
- **stop.\***：读 PID 终止。
- **status.\***：检查 PID 存活 + 端口监听 + 展示最近日志。
- **systemd**：`JenkinsConfig/isas.service`，支持 `systemctl start/stop/status isas`（Linux）。
- **Jenkins**（依《生成Jenkinsfile的提示词.md》）：Jenkinsfile 实现 停服->git pull(SSH)->装依赖/构建->启服；流水线 "Pipeline script from SCM"，脚本路径 `src/JenkinsConfig/Jenkinsfile`，SSH 远程仓库，每 30 分钟轮询 GitHub 触发；部署路径 `/opt/`。手工触发构建后让用户验证。
- **.gitignore**：含 `logs/`、`node_modules/`、`__pycache__/`、`venv/`、`dist/`、`*.pyc`；**不含** `data/`。
